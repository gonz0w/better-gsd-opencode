<task type="auto">
  <name>Task 1: Create audit command module with scanner, rubric, and estimator</name>
  <files>src/commands/audit.js</files>
  <action>
Create `src/commands/audit.js` following the existing command module pattern (see `src/commands/codebase.js` for reference).

**Scanner (decision point extractor):**
- Scan workflow `.md` files from the `workflows/` directory (resolved via `$BGSD_HOME` environment variable — use `process.env.BGSD_HOME` or resolve from `__dirname` relative paths up to the plugin root, following the same approach `command-enricher.js` uses)
- Also scan agent `.md` files from the `agents/` directory at the same location
- Use regex-based pattern matching to identify decision points in markdown. Decision indicators (from RESEARCH.md):
  - Routing/choice tables: `/\|\s*Condition\s*\|\s*(?:Meaning|Action|Route)\s*\|/i`, `/\|\s*(?:Route|Pattern)\s+[A-F]\b/i`
  - Decision conditionals: `/\bIf\s+.*(?:ask|offer|present|suggest|route|classify)\b/i`
  - Decision verbs: `/\b(?:choose|decide|determine|select|pick)\b/i`
  - Option presentation: `/options?:\s*\n\s*-/i`, `/(?:offer|present).*(?:options?|choice)/i`
  - Classification: `/\b(?:infer|classify|categorize|severity|priority)\b/i`
  - Rule application: `/\b(?:heuristic|rule|criteria)\b.*\b(?:apply|check|use)\b/i`
- For each match, extract surrounding context (5-10 lines before/after) to capture the decision's inputs and outputs
- Filter out false positives: procedural conditionals like "if file exists" are NOT decisions — only extract where the LLM must CHOOSE between alternatives or CLASSIFY input
- Assign each candidate a unique ID (e.g., `WF-progress-route-selection`, `AG-planner-tdd-detection`)
- Categorize each candidate into one of the 7 categories from RESEARCH.md: workflow-routing, model-selection, execution-mode, state-assessment, severity-classification, file-resolution, template-selection

**Rubric scorer (pure functions):**
- Implement `scoreCandidate(candidate)` as a pure function
- 3 critical criteria (ALL must pass for offloading):
  - `finite_inputs`: Decision has bounded set of possible inputs (enum or pattern-matchable)
  - `deterministic_output`: Same inputs always produce same output
  - `no_nlu_needed`: No natural language understanding required
- 4 preferred criteria (nice-to-have, scored 0-4):
  - `high_frequency`: Executes frequently (every plan or every session)
  - `low_complexity`: Implementation is simple (lookup table, <20 lines)
  - `existing_pattern`: Similar logic already exists in codebase
  - `low_blast_radius`: Changing affects few downstream consumers
- Apply rubric heuristically based on the decision category and extracted context:
  - `workflow-routing` and `state-assessment` categories default to passing all 3 critical criteria (finite inputs: counts/flags, deterministic: same state → same route, no NLU)
  - `severity-classification` passes for keyword-based patterns, fails for complex text assessment
  - `template-selection` passes for mechanical selection (commit type from files), fails for creative generation
- Return `{ passes, criteria, preferred_score, total_score, rationale }` where `rationale` explains the scoring for "keep in LLM" candidates

**Token estimator:**
- Implement `estimateTokenSavings(candidate)` as a pure function
- Token estimation categories (from RESEARCH.md):
  - `simple_lookup`: range [50, 100], midpoint 75 — command routing, file extension mapping
  - `conditional_chain`: range [200, 500], midpoint 350 — next-action routing, pattern selection
  - `multi_step_reasoning`: range [300, 800], midpoint 550 — deviation classification, severity inference
  - `context_overhead`: range [100, 300], midpoint 200 — prompt instructions for the decision
- Assign each candidate to a token category based on its decision category
- Estimate invocation frequency from workflow type: every-session (progress, resume, execute), per-phase (plan, research, discuss), rare (new-project, milestone)
- Calculate per-invocation and per-session savings: `savings = token_midpoint * frequency_per_session`

**Command handler (`cmdAuditScan`):**
- Accept `(cwd, args, raw)` following the standard pattern
- Resolve the plugin home directory for scanning workflows/agents
- Run scanner → rubric → estimator pipeline
- Return structured JSON with:
  - `candidates`: array of all identified decision points with rubric scores and token estimates
  - `offloadable`: candidates that pass the rubric (sorted by total_score descending)
  - `keep_in_llm`: candidates that fail the rubric (with rationale per candidate)
  - `summary`: { total_candidates, offloadable_count, keep_count, estimated_total_savings, savings_by_category }
- Use `output(result, raw)` for dual-mode output

Follow project conventions: `'use strict'`, 2-space indent, single quotes, `module.exports` at end, section dividers with Unicode box-drawing characters, `const { output, error, debugLog } = require('../lib/output')` imports.
